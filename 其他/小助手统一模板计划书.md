# 小助手统一模板计划书 2.0

## 一、项目背景

目前系统中存在多个小助手模块，包括：人设小助手、时间小助手、色色小助手、人物小助手、故事小助手和总结小助手。每个小助手都有自己的实现方式和运行机制，缺乏统一的模板和流程，导致维护和扩展困难。同时，需要引入大总管模块来统一管理API输入，提高系统效率。

## 二、目标

1. 统一所有小助手的模板和运行机制
2. 实现小助手的接入和输出连接
3. 允许用户自定义小助手向API输入的内容
4. 建立标准的小助手运行流程：提示词→API响应→分析→输出
5. 实现大总管模块，统一管理API输入，提高系统效率
6. 确保各个小助手与大总管的顺畅交互

### 3. 统一的运行流程

1. **输入阶段**：
   - 收集上下文信息
   - 收集用户输入的自定义提示词
   - 构建完整的提示词

2. **API调用阶段**：
   - 向大总管提交API输入
   - 接收大总管分发的响应

3. **分析阶段**：
   - 解析API响应
   - 提取关键信息
   - 进行逻辑处理

4. **输出阶段**：
   - 生成标准化的输出
   - 展示结果给用户

### 4. 大总管设计

#### 4.1 大总管职责
1. **输入管理**：收集和管理所有小助手的API输入
2. **输入合并**：分析输入内容，合并相同或相似的输入，减少API调用次数
3. **批量处理**：将合并后的输入批量发送到API，提高效率
4. **响应分发**：接收API响应，根据小助手ID分发给对应的小助手
5. **错误处理**：处理API调用错误，进行重试或通知小助手
6. **性能监控**：监控API调用性能，优化调用策略
7. **流程协调**：协调各个小助手的运行流程，确保系统顺畅运行

#### 4.2 大总管与小助手的交互流程
1. **输入提交**：小助手需要API调用时，通过统一接口向大总管提交输入
2. **输入队列**：大总管将输入加入输入队列，等待处理
3. **输入合并**：大总管定期检查队列，分析输入内容，合并相同或相似的输入
4. **批量发送**：大总管将合并后的输入批量发送到API
5. **响应接收**：大总管接收API返回的批量响应
6. **响应分发**：大总管根据小助手ID分发给对应的小助手
7. **结果处理**：小助手处理各自的响应并输出结果
8. **状态记录**：大总管记录API调用状态和性能指标

#### 4.3 大总管集成到小助手系统
1. **初始化**：系统启动时初始化大总管模块
2. **小助手注册**：各个小助手向大总管注册，提供API调用需求
3. **事件监听**：大总管监听故事事件，协调小助手运行
4. **输入处理**：大总管统一处理所有API输入，确保高效调用
5. **响应分发**：大总管将API响应分发给对应的小助手
6. **状态管理**：大总管管理整个系统的运行状态，确保稳定运行

#### 4.4 大总管优势
- **减少API调用**：合并相同输入，减少网络延迟
- **统一管理**：集中处理API调用，简化系统架构
- **错误处理**：统一处理错误，提高系统稳定性
- **性能优化**：监控API调用性能，优化系统运行
- **扩展性**：易于添加新的小助手，系统扩展性强

### 4. 各个小助手的具体实现

#### 4.1 人设小助手 (ProfileAssistant)
- **作用**：管理角色的描述、性格、背景、关系、场景描写示例、创作者笔记、标签、物品、事件和数值
- **读取内容**：角色信息、物品列表、事件列表、数值、故事档案
- **向大总管输入**：故事结束时发送角色信息和故事内容，输入标签：`角色信息`, `故事上下文`，输出标签：`日记`, `档案`, `数值`, `物品`, `事件`
- **运行逻辑**：监听故事结束事件，读取角色信息和故事内容，向大总管提交输入，接收更新内容，更新角色档案

#### 4.2 时间小助手 (TimeAssistant)
- **作用**：管理游戏世界的时间流逝，根据故事内容智能调整时间
- **读取内容**：世界时间设置、故事内容
- **向大总管输入**：每次生成故事时，输入标签：`故事上下文`，输出标签：`时间`
- **运行逻辑**：初始化世界时间，监听故事内容更新事件，分析时间信息，调整世界时间

#### 4.3 色色小助手 (EroticAssistant)
- **作用**：管理角色的色情内容和CG触发，检测关键词，根据角色数值触发场景，向故事小助手输入玩法信息
- **读取内容**：关键词配置、数值配置、角色数值、故事内容
- **向大总管输入**：故事内容，输入标签：`故事上下文`, `角色数值`，输出标签：`数值`
- **运行逻辑**：监听故事内容更新事件和聊天事件，分析关键词，检查数值阈值，生成相应的玩法信息，向故事小助手发送玩法信息

#### 4.4 人物小助手 (CharacterGeneratorAssistant)
- **作用**：根据描述快速生成临时角色，从描述中提取角色信息
- **读取内容**：用户输入的角色描述，故事内容，临时角色库
- **向大总管输入**：角色描述，故事内容，临时角色库，输入标签：`角色描述`, `故事上下文`，输出标签：`角色`
- **运行逻辑**：监听故事内容更新事件，读取角色描述和故事内容，向大总管提交输入，保存临时角色

#### 4.5 故事小助手 (StoryAssistant)
- **作用**：管理故事的剧情和发展，生成故事内容，生成剧情建议，接收并处理色色小助手发送的玩法信息
- **读取内容**：故事类型和风格设置、角色信息、故事内容、色色小助手发送的玩法
- **向大总管输入**：故事内容，角色信息，输入标签：`故事设置`, `角色信息`, `故事上下文`，输出标签：`故事内容`, `剧情建议`
- **运行逻辑**：监听用户输入和故事内容更新事件，接收色色小助手发送的玩法信息，根据玩法信息调整剧情走向，向大总管提交输入，输出生成的故事内容

#### 4.6 总结小助手 (SummaryAssistant)
- **作用**：管理故事卡片和生成故事日记，生成故事总结
- **读取内容**：故事内容、日记设置
- **向大总管输入**：故事内容，输入标签：`日记设置`, `故事上下文`，输出标签：`日记`
- **运行逻辑**：监听故事结束事件，读取故事卡片，向大总管提交输入，保存故事日记到故事档案

## 八、风险评估

1. **兼容性风险**：修改现有小助手可能影响现有功能
2. **性能风险**：统一模板可能增加代码复杂度，大总管模块可能增加系统负载
3. **用户体验风险**：更改界面可能需要用户适应
4. **系统集成风险**：大总管模块与现有小助手的集成可能出现问题
5. **API调用风险**：批量处理API输入可能导致响应延迟
6. **显示模式风险**：多种输出显示模式可能增加开发复杂度和维护成本
7. **编辑界面风险**：编辑小助手的显示功能可能与现有功能冲突

## 九、解决方案

1. **兼容性**：在修改现有小助手时，保持向后兼容，确保现有功能不受影响
2. **性能**：优化代码结构，减少不必要的计算，对大总管模块进行性能优化
3. **用户体验**：保持界面风格一致，提供清晰的操作指引，确保用户能够快速适应新界面
4. **系统集成**：详细规划大总管与小助手的集成方案，进行充分的测试
5. **API调用**：优化批量处理策略，确保API响应及时分发
6. **显示模式**：采用模块化设计，确保不同显示模式之间的切换流畅，减少开发复杂度
7. **编辑界面**：设计清晰的编辑界面，确保与现有功能无缝集成，提供良好的用户体验
